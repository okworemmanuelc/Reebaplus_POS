# Reebaplus POS вЂ” Pivot Plan (revised after user decisions)

> Revised 2026-05-26 after user resolved all 10 open questions from the first draft. Decisions are captured in this revision and logged in [BUILD_LOG.md](BUILD_LOG.md) Session 1. The master plan ([reebaplus_master_plan.md](reebaplus_master_plan.md)) was updated to match (В§1.1, В§2.3, В§7.1, В§16.5, new В§16.11).

## 0. Summary

The codebase has a mature offline-first foundation (Drift schema v12, 22 DAOs, robust sync layer, ~42 screens) but it was built around a single-owner-per-business model. The master plan calls for a data-driven, multi-role, multi-store architecture with a brand-new Funds Register subsystem and a renamed vocabulary throughout.

The good news: the master plan's "critical fix first" (В§2.3) вЂ” dropping `users_role_tier_check` вЂ” is **already done**. Commit `38ea06b` and Supabase migration [0041_remove_staff_management.sql](supabase/migrations/0041_remove_staff_management.sql) wiped the old staff/role/invite system from both the Drift schema and the cloud. The current Users table at [app_database.dart:104-135](lib/core/database/app_database.dart#L104-L135) has no role columns. The pivot starts from a clean slate.

The hard parts: (1) a whole permission/role subsystem (4 default roles, ~25 permission keys, per-role toggles, per-role limits) must be designed and seeded on business creation; (2) the Funds Register is genuinely new вЂ” 3 new tables plus the Open Day / Close Day flows and integration into every money path; (3) a global rename pass (Warehouse в†’ Store, Customer Group в†’ Price Tier, Dashboard в†’ Home, Settings в†’ CEO Settings, Crate Group в†’ Crate Size Group, Purchases в†’ Shipments) will touch hundreds of files. Realistic execution: ~32 focused sessions.

The "one email в†’ many businesses" feature has been **scoped down**: Phase 1 = one user / one business; Phase 2 = switch-business picker UI. The `user_businesses` table is still built in Phase 1 so the database supports it.

Sync leak audit (В§6) is clean вЂ” every flagged write is either properly paired with `enqueueUpsert` or is a legitimate exception with an in-code justification. The codebase has more documented exceptions than CLAUDE.md В§5 currently names вЂ” that doc should be updated during the pivot.

No nasty surprises in the existing code. The DAO/sync contract is well-followed. Most existing screens have rough equivalents in the master plan вЂ” they need restructure, not rebuilds from zero.

## 7. Risk register

- **Q4 product price column drop is destructive.** All four legacy price columns are dropped with no data migration. Existing product data on test devices loses its prices entirely. User has confirmed they'll manually re-enter prices after the migration. **The session running the v15 migration must re-confirm this with the user before executing.** Add it as a hard checkpoint in В§8.
- **Existing customer / order / inventory data.** The codebase is in active development per memory ([project_role_refactor.md], [project_sync_fk_bug.md] both landed on main 2026-05-22). Renames and column drops must use Drift's `TableMigration` (new-table / copy / drop / rename) to preserve rows. Cloud-side: matching migrations needed for every Drift migration. Plan a data-preservation pass before each restructure step.
- **Customer group в†’ price tier transition.** Existing rows may have `customer_group = 'distributor'` (migrate to `'wholesaler'`) or `'walk_in'` (hard-delete вЂ” these shouldn't exist). The new CHECK constraint will reject the old values. Test on a populated DB before rolling.
- **Funds Register integration touches every money path.** Sales, refunds, expenses, supplier payments, wallet top-ups all need to debit/credit a funds account from now on. Easy to miss a path during the build. The pivot order in В§8 splits this into a dedicated session (step 17) to wire every path with care after the checkout UI is done.
- **The Drift schema audit and sync FK fix.** Memory ([project_sync_fk_bug.md]) flags a latent `statement_timeout` risk on `pos_pull_snapshot`. Adding ~9 new synced tables will lengthen every snapshot pull. The post-create loop in [app_database.dart:1528-1532](lib/core/database/app_database.dart#L1528-L1532) auto-adds `(business_id, last_updated_at)` indexes for new synced tables вЂ” good вЂ” but pre-emptively measure snapshot time after the new tables are seeded with realistic data.
- **Role guards in EVERY screen.** Master plan В§30.1 / CLAUDE.md hard rule #6 / coding rule #1. The current sidebar has zero guards. The risk is shipping screens with the guards "to be added later" and forgetting. Add a single `RoleGuard(permissionKey: 'x.y')` widget early and wrap every action with it as a discipline.
- **The migration window.** v12 already touched users + dropped business_members + invites. Adding the pivot's v13+ migrations on top requires careful sequencing вЂ” Drift's `MigrationStrategy.onUpgrade` uses cumulative `if (from < N)` blocks. Each new migration adds another block; mistakes here are hard to roll back.
- ~~**CLAUDE.md В§5 vs reality.**~~ **Resolved 2026-05-26.** В§5 has been updated to enumerate all six legitimate sync-write exceptions. See В§6 for the full list and CLAUDE.md for the canonical wording.
