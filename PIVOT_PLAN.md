# Reebaplus POS — Pivot Plan (revised after user decisions)

> Revised 2026-05-26 after user resolved all 10 open questions from the first draft. Decisions are captured in this revision and logged in [BUILD_LOG.md](BUILD_LOG.md) Session 1. The master plan ([reebaplus_master_plan.md](reebaplus_master_plan.md)) was updated to match (§1.1, §2.3, §7.1, §16.5, new §16.11).

## 0. Summary

The codebase has a mature offline-first foundation (Drift schema v12, 22 DAOs, robust sync layer, ~42 screens) but it was built around a single-owner-per-business model. The master plan calls for a data-driven, multi-role, multi-store architecture with a brand-new Funds Register subsystem and a renamed vocabulary throughout.

The good news: the master plan's "critical fix first" (§2.3) — dropping `users_role_tier_check` — is **already done**. Commit `38ea06b` and Supabase migration [0041_remove_staff_management.sql](supabase/migrations/0041_remove_staff_management.sql) wiped the old staff/role/invite system from both the Drift schema and the cloud. The current Users table at [app_database.dart:104-135](lib/core/database/app_database.dart#L104-L135) has no role columns. The pivot starts from a clean slate.

The hard parts: (1) a whole permission/role subsystem (4 default roles, ~25 permission keys, per-role toggles, per-role limits) must be designed and seeded on business creation; (2) the Funds Register is genuinely new — 3 new tables plus the Open Day / Close Day flows and integration into every money path; (3) a global rename pass (Warehouse → Store, Customer Group → Price Tier, Dashboard → Home, Settings → CEO Settings, Crate Group → Crate Size Group, Purchases → Shipments) will touch hundreds of files. Realistic execution: ~32 focused sessions.

The "one email → many businesses" feature has been **scoped down**: Phase 1 = one user / one business; Phase 2 = switch-business picker UI. The `user_businesses` table is still built in Phase 1 so the database supports it.

Sync leak audit (§6) is clean — every flagged write is either properly paired with `enqueueUpsert` or is a legitimate exception with an in-code justification. The codebase has more documented exceptions than CLAUDE.md §5 currently names — that doc should be updated during the pivot.

No nasty surprises in the existing code. The DAO/sync contract is well-followed. Most existing screens have rough equivalents in the master plan — they need restructure, not rebuilds from zero.

### 1.2 New tables to add

All from master plan §2.4 unless noted. All are synced tenant tables and must be added to `_syncedTenantTables` in [app_database.dart:1389](lib/core/database/app_database.dart#L1389) (see §1.5).

| `permissions` | Master list of permission keys (see master plan §2.5). Global, not per-business — lives in `system_config` or its own non-tenant table. | No (global) | `PermissionsDao` |

| `funds_days` | Per-(store, date) row holding the day's state: opened/closed timestamps, who opened/closed. Master plan §23.4 / §23.6. | Yes | `FundsDaysDao` |
